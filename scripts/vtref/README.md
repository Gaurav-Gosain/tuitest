# Differential testing against a reference VT

tuitest reports what a program rendered, so the honesty of every result it has
ever produced rests on `internal/vt` being right. Our own unit tests cannot
establish that: they encode the same assumptions the emulator does. This
directory holds the tooling that checks the emulator against somebody else's.

## The reference

`dump.mjs` drives **ghostty-vt compiled to wasm** from node and prints the
screen as JSON. The wasm is not vendored here; it is recoverable from the `sip`
repository's history:

```
git -C /path/to/sip show ec3b444^:static/ghostty-web/ghostty-vt.wasm > ref/ghostty-vt.wasm
cp scripts/vtref/dump.mjs ref/
```

It was chosen because it exposes grid-scoped cell reads (`ghostty_terminal_grid_ref`
plus `ghostty_cell_get` / `ghostty_grid_ref_style`), so character, width,
colours and attributes can all be read back per cell, and because it had
already been validated against a corpus in that project. Driving the wasm from
Go would have meant adding a wasm runtime to tuitest's dependency graph for a
test-only tool, so the reference runs in node and the comparison happens in Go
over JSON.

### Limitations of the reference

- **Erase does not carry the background colour.** This build reports cells
  erased by ED/EL under a non-default background as having the default style.
  tmux, cross-checked as a third opinion, agrees with tuitest that the
  background is kept (that is xterm's "background colour erase"), so this class
  of difference is a reference artefact and is deliberately not "fixed".
- **NUL and DEL** consume a cell in this build; tmux and tuitest both ignore
  them. tuitest follows tmux.
- **C1 controls arriving as UTF-8** (U+0084, U+0085) are given a cell by the
  reference and dropped by tuitest. No real program emits them.
- Only the active screen is compared. Scrollback is out of scope here.

## Running it

```
TUITEST_VT_REF=/path/to/ref go test -run TestVTDifferential .
```

The test skips when the variable is unset or node is missing, so it never
blocks a normal `go test ./...`. **It is a discovery tool, not the regression
net**: every divergence it finds is copied down into an ordinary unit test in
`internal/vt/fidelity_test.go` with the reference's answer hard-coded, so the
fix stays enforced on a machine that has neither node nor the wasm.

## The corpus

`testdata/vtcorpus/*.vt` holds two kinds of input.

- `prog_*.vt` are real recordings: the raw PTY output of htop, btop, less, vim,
  tmux and fish. Regenerate with `record.py`, which sizes a PTY, records for a
  fixed time and kills the child's process group afterwards:

  ```
  python3 scripts/vtref/record.py out.bin 80 24 2.0 -- htop
  ```

- Everything else is adversarial, generated by `gencorpus.py`: scroll regions,
  insert and delete line and character, erase in line and display in every
  mode, tab stops, origin mode, autowrap and reverse wrap, character sets, save
  and restore cursor, the alternate screen with and without cursor save, mouse
  and bracketed-paste toggles, wide runes and combining marks, and sequences
  with absurd, empty or truncated parameters.

  ```
  python3 scripts/vtref/gencorpus.py testdata/vtcorpus
  ```
